rAF Position 04 / 08

Where Does rAF Fire?

rAF callbacks run at the start of the render phase — before Style, Layout, or Paint begins. This gives you a guaranteed slot right at the top of each frame.

Any DOM mutations you make inside an rAF callback are batched by the browser and flushed in the very next Style/Layout/Paint pass — never split across frames.

rAF = first thing that runs in each render frame

Comparison 06 / 08

rAF vs setTimeout

setTimeout / setInterval
  • Fires from the task queue — not the render phase
  • Timing is approximate; can fire mid-frame or be delayed
  • May run multiple times per frame, or skip frames entirely
  • No awareness of the display's actual refresh rate
  • Keeps running even when the tab is hidden (wastes CPU)
requestAnimationFrame
  • Fires at the render checkpoint — perfectly synced
  • Exactly once per frame, every frame
  • Matches the display's actual refresh rate (60/120/144 Hz)
  • Receives a precise DOMHighResTimeStamp argument
  • Automatically pauses when the tab is hidden
For visual animation, rAF is always the right choice
In Practice 07 / 08

Using rAF
Correctly

  • Schedule with requestAnimationFrame(fn)
  • Use the timestamp argument for smooth motion — never Date.now()
  • Re-schedule inside the callback to keep the loop running
  • Store the return ID and cancel with cancelAnimationFrame(id)
let animId; let startTime = null; function animate(timestamp) { // Re-schedule at the start of each frame animId = requestAnimationFrame(animate); if (!startTime) startTime = timestamp; const elapsed = timestamp - startTime; // Drive position with elapsed time el.style.transform = `translateX(` + (elapsed * 0.05) + `px)`; } animId = requestAnimationFrame(animate); // Stop cleanly when done cancelAnimationFrame(animId);
timestamp is DOMHighResTimeStamp in milliseconds — high precision
Key Takeaways 08 / 08

What to Remember

  1. 01
    rAF fires before Style, Layout & Paint It's the first thing in each frame's render phase. DOM changes made here are batched into the same paint.
  2. 02
    Once per frame, synced to the display At 60 Hz: every 16.67 ms. At 120 Hz: every 8.33 ms. It matches the actual hardware refresh rate.
  3. 03
    Drive animation with the timestamp, not Date.now() The DOMHighResTimeStamp argument is far more precise and consistent across frames.
  4. 04
    Free CPU when the tab is hidden rAF automatically pauses when the page isn't visible — saving battery and CPU with zero extra code.
